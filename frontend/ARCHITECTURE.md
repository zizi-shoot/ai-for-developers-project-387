# Архитектурные подходы проекта

## Разделение ответственности

- Разделяй код на слои по ответственности: запросы API, бизнес-логика, вспомогательные утилиты и
  UI-компоненты.
- Слой запросов API отвечает только за взаимодействие с внешними сервисами: формирование запросов,
  вызов клиента, валидацию и нормализацию ответов.
- Слой бизнес-логики содержит правила предметной области, подготовку данных для сценариев и
  координацию операций между API, состоянием и UI.
- Слой вспомогательных утилит содержит переиспользуемые чистые функции и технические хелперы без
  знания о конкретных пользовательских сценариях.
- Слой UI-компонентов должен только отображать данные и содержать исключительно UI-логику: обработку
  пользовательских действий, локальное состояние отображения и визуальные состояния.
- Хранение клиентского состояния должно быть либо в локальных компонентах, либо в Pinia store.
- Хранение серверного состояния должно быть в TanStack Query.

## Структура проекта по Feature Sliced Design (FSD)

### Порядок слоёв

- Используй FSD слои: `shared -> entities -> features -> widgets -> pages -> app`
- Верхние слои могут импортировать только нижележащие слои.
- Избегай циклических зависимостей.
- Создавай `features`, `widgets` и `entities` только для сущностей, которые переиспользуются на
  нескольких страницах. Логику, доменные объекты и UI, которые нужны только одной странице, оставляй
  в папке этой страницы внутри `pages`.

### Структура папок

- Держи доменный код внутри сегмента его слоя (`api`, `model`, `ui`, `lib`).
- Делай файлы небольшими и сфокусированными на одной ответственности.
- Используй один публичный вход на слайс: `index.ts`.

### Правило публичного API

- `index.ts` разрешён только как тщательно отобранный публичный API.
- Не используй wildcard-экспорты (`export * from ...`) в файлах публичного API.
- Экспортируй только стабильные символы, нужные другим слоям.
