# Agents guidelines

## Стек приложения

- Vue 3
- Vite — сборщик приложения
- PrimeVue — UI-библиотека
- @stoplight/prism-cli  — эмуляция API по контракту
- dayjs — работа с датами
- Vue Testing Library — unit-тестирование Vue-компонентов через пользовательские сценарии
- Testing Library user-event — симуляция реалистичных действий пользователя в unit-тестах
- Tanstack Vue Query - хранение серверного состояния на клиенте

## Архитектура

- разделяй отвественность по слоям: слой хранения данных, бизнес-логика в отдельных функциях или vue-composables, vue-компоненты для слоя представления
- структура проекта описана в @ARCHITECTURE.md

## Тестирование

- для unit-тестов используй Vitest, `@testing-library/vue` и `@testing-library/user-event`; тесты размещай рядом с проверяемым кодом или в `src/__tests__` с суффиксом `.spec.ts`
- для моков API в e2e используй `mockApiRoute` из `e2e/mock-api.ts`, чтобы preflight и CORS-заголовки обрабатывались единообразно
- проверяй поведение компонента с точки зрения пользователя: рендерь компонент через `render`, ищи элементы через `screen` по роли, label или видимому тексту
- для действий пользователя создавай `const user = userEvent.setup()` в каждом тесте и вызывай `await user.click(...)`, `await user.type(...)`, `await user.keyboard(...)`, `await user.selectOptions(...)` или `await user.upload(...)`
- не изменяй DOM напрямую через `element.value = ...` и `dispatchEvent`; `fireEvent` используй только для низкоуровневых событий, которые не покрывает `user-event`
- не тестируй приватную реализацию компонента, внутренние refs и CSS-классы, если они не являются частью пользовательского поведения
- для маршрутизации, Pinia и PrimeVue подключай в тесте минимальные плагины и mocks, необходимые конкретному сценарию
- перед завершением frontend-задачи запускай `pnpm test:unit`; если изменения затрагивают пользовательские сценарии в браузере, также запускай `pnpm test:e2e`
