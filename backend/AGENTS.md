# Backend Guidelines

## Стек

- go
- typespec

## Общий подход

- Реализуй бэкенд как модульный монолит.
- Исходи из рамок `../GOAL.md`: один заранее заданный read-only владелец, гостевое бронирование без учётных записей, регистрации и авторизации.
- Используй простую слоистую архитектуру: `HTTP handler -> service -> repository`.
- Группируй код по бизнес-возможностям: `booking`, `eventtype`, `availability`, `owner`.
- Предпочитай стандартную библиотеку Go. Не добавляй HTTP-фреймворк, router, validator или генератор ID без согласования с пользователем.

## Границы слоёв

- HTTP handler отвечает только за HTTP: path-параметры, JSON, headers, status codes и преобразование ошибок.
- Service содержит бизнес-правила и не зависит от `net/http`, JSON и конкретного хранилища.
- Repository отвечает за хранение, конкурентную безопасность и атомарные операции над данными.
- `main.go` только собирает зависимости, настраивает HTTP server и управляет его жизненным циклом.
- Вводи интерфейсы на границах, где это даёт заменяемость или тестируемость: repository и источник текущего времени (`Clock`).

## Контракт API

- Считай `../main.tsp` единственным источником истины для routes, DTO, status codes и формата ошибок.
- Перед изменением API проверяй согласованность `main.tsp`, Go-реализации и frontend-клиентов.
- Возвращай JSON с корректным `Content-Type`. Пустые коллекции кодируй как `[]`, а не `null`.
- Возвращай доменные и клиентские ошибки в едином виде `{"code":"...","message":"..."}`.
- Используй стабильные машиночитаемые коды, например `INVALID_REQUEST`, `INVALID_SLOT`, `EVENT_TYPE_NOT_FOUND`, `BOOKING_CONFLICT`.
- Различай: невалидный запрос — `400`, отсутствующий ресурс — `404`, конфликт бронирования — `409`.

## Бизнес-правила

- Бэкенд — единственный источник истины о допустимости бронирования. Не доверяй предварительной проверке фронтенда.
- Слоты не хранятся, а вычисляются из расписания, текущего времени, длительности типа события и существующих бронирований.
- Формируй слоты на 14 календарных дней от начала текущей UTC-даты, в расписании 09:00–18:00 UTC.
- Генерируй начала слотов с шагом 15 минут. Событие должно целиком закончиться не позже 18:00 UTC.
- Не возвращай прошедшие слоты текущего дня.
- Используй UTC в бизнес-логике. Принимай и возвращай даты в RFC 3339.
- Считай занятым весь интервал `startsAt + durationMinutes`, а не только момент начала.
- Рассматривай интервалы как полуоткрытые `[start, end)`. Два интервала пересекаются, если `newStart < existingEnd && existingStart < newEnd`.
- Запрещай пересечения со всеми бронированиями, независимо от типа события.
- При создании бронирования заново проверяй существование типа события, окно записи, рабочие часы, сетку слотов и отсутствие пересечений.

## Конкурентность и хранение

- Храни типы событий и бронирования в памяти. Потеря данных при перезапуске допустима.
- Защищай общее состояние через `sync.RWMutex`. Не возвращай наружу изменяемые внутренние slices или maps.
- Проверка пересечения и добавление бронирования должны быть одной атомарной операцией под одной write-lock. Не разделяй их на независимые repository-вызовы.
- Учитывай, что in-memory хранилище не поддерживает несколько экземпляров сервиса с общим состоянием.

## Валидация и безопасность

- Разделяй синтаксическую, структурную и бизнес-валидацию.
- Отклоняй невалидный JSON, неизвестные поля, лишние JSON-значения после объекта и пустые обязательные строки.
- Ограничивай размер request body.
- Проверяй `durationMinutes` по контракту, email — на разумный формат, `startsAt` — на RFC 3339, UTC и 15-минутную сетку.
- Не логируй секреты и лишние персональные данные гостей.
- Учитывай CORS для локального frontend origin и корректно обрабатывай preflight `OPTIONS`. Не открывай origins шире необходимого.

## HTTP server и наблюдаемость

- Настрой таймауты HTTP server и graceful shutdown по сигналу завершения.
- Добавь middleware для восстановления после panic и контекстного логирования запросов.
- Логируй как минимум method, path, status code, duration и request ID.
- Не отдавай клиенту внутренние ошибки, stack traces или детали инфраструктуры.

## Тестирование

- Пиши table-driven tests для бизнес-правил и HTTP handlers.
- Тестируй handlers через `net/http/httptest`, не запуская реальный сервер без необходимости.
- Используй фиксированный `Clock` в тестах даты и слотов.
- Обязательно покрой: границы рабочего дня, прошедшие слоты, 14-дневное окно, событие после 18:00, пересечение разных типов событий, соседние непересекающиеся интервалы, сортировку upcoming bookings и HTTP-ошибки.
- Добавь конкурентный тест: из двух одновременных попыток занять один слот успешна только одна.
- Перед завершением запускай `go test ./...`, `go test -race ./...` и доступную проверку TypeSpec-контракта.
