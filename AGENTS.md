# Repository Guidelines

## Структура проекта

- `frontend/` — Vue-приложение.
- `backend/` — Go backend.
- также в корне проекта лежат файлы для создания и генерации typespec-спецификации.

## Рабочий процесс

- Общайся с пользователем на русском языке. Имена переменных, функций, пакетов и другие кодовые идентификаторы оставляй на английском.
- Перед изменениями изучи текущие файлы проекта и следуй уже выбранным паттернам.

## Документация и зависимости

- Описывай сложные участки кода комментариями на русском языке.
- Комментарий должен отвечать на вопрос "Что делает", а не "Как делает".
- Для вопросов про библиотеки, фреймворки, SDK, API, CLI-инструменты и cloud-сервисы используй `ctx7` CLI для актуальной документации.
- Сначала получи идентификатор библиотеки:

```bash
npx ctx7@latest library <name> "<полный вопрос пользователя>"
```

- Затем запроси документацию по выбранному ID:

```bash
npx ctx7@latest docs <libraryId> "<полный вопрос пользователя>"
```

- Не выполняй больше 3 команд `ctx7` на один вопрос.
- Не включай в запросы к `ctx7` секреты, API-ключи, пароли или credentials.
- Если `ctx7` завершился quota error, сообщи пользователю и предложи выполнить `npx ctx7@latest login` или задать `CONTEXT7_API_KEY`.
- Если `ctx7` падает из-за DNS, host resolution или network/fetch error внутри песочницы Codex, повтори команду вне стандартной песочницы.
- Активно предлагай готовые библиотеки для типовых задач, но перед добавлением новой зависимости спроси разрешение и кратко объясни пользу и альтернативы.

## Проверки

- Перед завершением задачи запускай все релевантные доступные проверки.
- Если проверка отсутствует, не выдумывай команду. Явно сообщи, что такая проверка пока недоступна.

## Безопасность

- Строго фиксируй версии зависимостей
- Любые ключи, секреты и прочие чувствительные данные должны лежать в `.env`-файлах, которые не должны попадать в коммит

## Git

- После завершенной задачи и успешных проверок сделай коммит, если пользователь не попросил иначе.
- Используй Conventional Commits: префикс на английском, описание на русском.
- Пиши краткие и понятные описания
- Примеры:
  - `feat(auth): добавил форму входа`
  - `fix(auth): исправил обработку ошибки авторизации`
  - `docs(agents): описал правила для агентов`
- **PR**:
  - Указывай цель, краткое описание изменений, связанные задачи, план тестирования. Держи набор изменений сфокусированным.
  - В заголовках PR не используй служебные префиксы вроде `[codex]`;
  - формулируй название как краткое существительное или noun phrase, например `Настройка правил проекта`.
