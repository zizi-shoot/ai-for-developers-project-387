# Описание приложения

В проекте есть две условные роли:

- **Владелец календаря**
- **Гости**

При этом в проекте нет регистрации и авторизации.

Владелец календаря — один заранее заданный профиль. Этот профиль по умолчанию используется в админской части.

Гость бронирует слоты без создания аккаунта и без входа в систему.

## Владелец календаря может

- Создавать типы событий.
- Для каждого типа события задавать:
    - `id`
    - название
    - описание
    - длительность в минутах
- Просматривать страницу предстоящих встреч, где в одном списке показаны бронирования всех типов событий.

## Гость может

- Посмотреть страницу с видами брони, где доступны:
    - название
    - описание
    - длительность
- Выбрать тип события.
- Открыть календарь.
- Выбрать свободный слот в ближайшие 14 дней.
- Создать бронирование на выбранный слот.

## Правило занятости

На одно и то же время нельзя создать две записи, даже если это разные типы событий.

## Окно записи по умолчанию

- Доступные слоты формируются на 14 дней, начиная с текущей даты.
- Гость может записаться только на свободный слот из этого окна.

## Требования к бэкенду

- Бэкенд предоставляет API по typspec-спецификации контракта.
- Основные бизнес-правила бронирования реализуются на стороне бэкенда.
- Отдельная база данных не нужна. Достаточно простого хранилища в памяти: после перезапуска сервиса данные могут сбрасываться.
